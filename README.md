# 📈 NIFTY.AI — Stock Prediction Platform

AI-powered web platform that forecasts the **next trading day's closing price** for all NIFTY 50 stocks using a hand-engineered technical-indicator feature set and a Multi-Layer Perceptron (Neural Network) regressor — fully automated with a daily data-refresh and retraining pipeline via GitHub Actions.

**Live demo:** [stock-prediction-platform-five.vercel.app](https://stock-prediction-platform-five.vercel.app)

> ⚠️ **Disclaimer:** This project is for educational and informational purposes only. Predictions are generated by a statistical model trained on historical price data and should **not** be treated as financial or investment advice. Always do your own research before making investment decisions.

## Overview

**NIFTY.AI** is a full-stack stock market prediction platform built around the **NIFTY 50** index (the 50 largest, most liquid companies listed on the National Stock Exchange of India). The platform:

1. Pulls and maintains a historical OHLCV (Open, High, Low, Close, Volume) dataset for every NIFTY 50 constituent stock.
2. Engineers a large set of technical-analysis features (moving averages, RSI, MACD, Bollinger Bands, ATR, Stochastic Oscillator, candlestick patterns, calendar effects, lag features, rolling statistics, etc.) from raw price data.
3. Trains a separate **MLPRegressor** (feed-forward neural network) **per stock** to predict the next day's closing price.
4. Converts the predicted price into an expected return, a confidence score, and a discrete trading **signal** (`STRONG BUY` / `BUY` / `HOLD` / `SELL` / `STRONG SELL`).
5. Serves these pre-computed predictions through a lightweight Express.js REST API.
6. Renders the predictions in a polished, dark-themed React dashboard with interactive charts.
7. Refreshes the dataset and retrains all 48 models **automatically every day** using a scheduled GitHub Actions workflow — no manual intervention required.

```
   Yahoo        │   update.py             │
   Finance  ───▶│   (fetch latest OHLCV)  │
   (yfinance)   └────────────┬────────────┘
                              │ appends to
                              ▼
                 backend/Nifty 50.csv
                              │
                              ▼
                ┌─────────────────────────┐
                │   train_stocks.py       │
                │   - feature engineering │
                │   - train MLPRegressor  │
                │     per stock           │
                │   - generate signal     │
                └────────────┬────────────┘
                              │ writes
                              ▼
                     predictions.json
                              │
                              ▼
                ┌─────────────────────────┐
                │   service.js (Express)  │
                │   GET  /api/stocks      │
                │   POST /api/predict     │
                └────────────┬────────────┘
                              │ HTTP / JSON
                              ▼
                ┌─────────────────────────┐
                │   React + Vite frontend │
                │   (Dashboard UI)        │
                └─────────────────────────┘
```

The heavy lifting (data fetch + model training) is **not** done on every API request. Instead, it runs once a day on a schedule, and the API simply serves the latest cached `predictions.json`. This keeps the API extremely fast and avoids re-training a neural network on every page load.

## Architecture

The project is a classic **three-tier** application, split into two independently runnable workspaces (`backend/` and `frontend/`), connected by a thin REST API:

1. **Data & ML layer (Python)** — Python scripts in `backend/` handle everything related to data acquisition (`update.py`) and model training/inference (`train_stocks.py`, `prediction_model.py`). These are run **offline / on a schedule**, not on-demand per HTTP request.
2. **API layer (Node.js / Express)** — `service.js` exposes two simple REST endpoints that read from the static `predictions.json` and the stock symbol list. It does **not** invoke Python at request time in the current implementation — predictions are pre-computed and just looked up.
3. **Presentation layer (React)** — A single-page dashboard (`Dashboard.jsx`) lets the user pick a stock, fires a request to the backend, and visualizes the result (price metrics, a BUY/SELL/HOLD signal badge, a forecast chart, and a plain-English analysis blurb).

During local development, Vite's dev server proxies any request to `/api/*` to `http://localhost:3000` (see `vite.config.js`), so the frontend and backend can run on separate ports without CORS friction.

## Machine Learning Pipeline

### Dataset

- Source file: `backend/Nifty 50.csv` (~129,000 rows).
- Covers **48 NIFTY 50 constituent symbols** (e.g. `RELIANCE.NS`, `TCS.NS`, `INFY.NS`, `HDFCBANK.NS`, etc. — see [`stockNames.route.js`](backend/routes/stockNames.route.js) for the full list), with daily history going back to **2015**.
- Columns: `Symbol, Date, Close, High, Low, Open, Volume`.
- The dataset is kept current by `update.py`, which:
  - Loads the existing CSV and finds the most recent date already stored.
  - Downloads only the missing days (incremental fetch, not a full re-download) for each symbol via `yfinance.download()`, falling back to `yfinance.Ticker().history()` if the primary call returns empty.
  - Retries failed symbol downloads up to 3 times with a backoff delay (Yahoo Finance can rate-limit/throttle bulk requests).
  - Adds a small delay between symbols to stay polite to the Yahoo Finance endpoint.
  - Deduplicates on `(Symbol, Date)`, sorts, and writes the combined dataset back to disk.

### Feature Engineering

`create_features()` (duplicated identically in both `train_stocks.py` and `prediction_model.py`) takes raw OHLCV rows and engineers **dozens of technical indicators**, grouped by symbol so that rolling/lag calculations never leak across different stocks. Highlights include:

| Category | Features |
|---|---|
| **Lagged price action** | `Prev_Open/High/Low/Close`, `Close_Lag1/2/3`, `Return_Lag1/2/3` |
| **Trend (Moving Averages)** | `SMA_20/50/100`, `EMA_9/12/20/26/50`, `Close_vs_EMA9`, `Close_vs_EMA20` |
| **Momentum** | `RSI_14`, `RSI_Change`, `RSI_MA5`, `RSI_Overbought`/`RSI_Oversold` flags, `Stoch_K`/`Stoch_D`/`Stoch_Cross`, `Williams_R`, `CCI_20` |
| **Trend strength** | `MACD`, `MACD_Signal`, `MACD_Histogram` |
| **Volatility** | `BB_Upper/Lower/Width/Percent` (Bollinger Bands), `ATR_14`, `ATR_Percent`, `Rolling_Volatility_5/10/20` |
| **Volume** | `Volume_MA10`, `Volume_Ratio` |
| **Candlestick structure** | `Body_Size`, `Body_Pct`, `Upper_Wick`, `Lower_Wick`, `Is_Bullish`, `Intraday_Position`, `High_Low_Spread`/`_Pct` |
| **Gaps** | `Gap`, `Gap_Pct`, `Gap_Up`, `Gap_Down` |
| **Calendar effects** | `Day_of_Week`, `Month`, `Quarter`, `Week_of_Year`, `Is_Month_Start`, `Is_Month_End` |
| **Rolling extremes** | `Rolling_Max_5/10/20`, `Rolling_Min_5/10/20`, `Close_vs_High_{w}`, `Close_vs_Low_{w}` |
| **Targets** | `Target_Return` (next-day % return), `Tomorrow_Close` (next-day closing price — the actual regression target) |

Rows with `NaN` values (introduced by rolling/lag windows at the start of each stock's history) are dropped before training, **except** for the most recent row per stock, which is preserved separately (`prediction_row`) so the model can still generate a forecast for "today → tomorrow" even though its target is unknown.

### Model

For **each stock independently**:

1. Features and target (`Tomorrow_Close`) are split chronologically — the first **80%** of rows for training, the last **20%** for testing (a time-respecting split, not a random shuffle, since this is a time series).
2. Both `X` and `y` are standardized with separate `StandardScaler` instances.
3. A `sklearn.neural_network.MLPRegressor` is trained with the architecture:
   ```python
   MLPRegressor(
       hidden_layer_sizes=[256, 128, 64, 32, 16],
       early_stopping=True,
       random_state=42
   )
   ```
   — a 5-hidden-layer feed-forward network that funnels down from 256 to 16 neurons, with early stopping enabled to reduce overfitting.
4. The trained model predicts on the held-out test set to compute evaluation metrics, and separately predicts on the **latest available row** of features to forecast tomorrow's close.
5. Predictions are inverse-transformed back to the original price scale.

**Evaluation metrics computed per stock:**

- `rmse` — Root Mean Squared Error of test-set predictions.
- `r2` — R² score of test-set predictions (also reused as a 0–100 **"Confidence"** score).
- `DirectionAccuracy` — whether predicted vs. actual return signs matched.

### Signal Generation

The predicted price is converted into an **expected % return** vs. the current price, which is then bucketed into a discrete trading signal:

| Expected Return | Signal |
|---|---|
| `≥ 2%` | 🟢 **STRONG BUY** |
| `≥ 0.5%` | 🟢 **BUY** |
| `> -0.5%` | ⚪ **HOLD** |
| `> -2%` | 🔴 **SELL** |
| `≤ -2%` | 🔴 **STRONG SELL** |

`train_stocks.py` also attaches a short, templated **plain-English analysis string** for each signal class (e.g. *"The model identifies strong bullish momentum supported by recent market trends..."*), which the frontend's `AnalysisCard` displays — falling back to its own locally-generated text if the field is missing.

All 48 results are aggregated into a single `predictions.json` file at the repository root, keyed by stock symbol.

> 💡 `prediction_model.py` is a standalone CLI variant of the same pipeline for a **single** symbol (`python prediction_model.py RELIANCE.NS`), useful for debugging or ad-hoc predictions without retraining every stock.

## Backend (API)

The backend is a minimal **Express.js** server (`backend/service.js`) — it does not call Python at request time; it simply serves pre-computed JSON.

```js
app.use("/api", predictionRoutes);
app.use("/api", stockNames);
app.listen(3000);
```

### Endpoints

#### `GET /`
Health check.
```json
{ "status": "ok" }
```

#### `GET /api/stocks`
Returns the static list of all supported NIFTY 50 symbols (hardcoded for fast response — see note below).

```json
["ADANIENT.NS", "ADANIPORTS.NS", "APOLLOHOSP.NS", "...", "WIPRO.NS"]
```

> **Note:** The route originally parsed the full `Nifty 50.csv` (~129k rows) on every request to extract unique symbols, which was too slow. It was switched to a hardcoded list of symbols for instant responses — the CSV-based logic is left in the source as a commented-out reference.

#### `POST /api/predict`
Looks up a stock's latest pre-computed prediction from `predictions.json`.

**Request body:**
```json
{ "stock": "RELIANCE.NS" }
```

**Response:** see [Sample API Response](#sample-api-response) below.

**Error response** (unknown symbol):
```json
{ "error": "Stock not found" }
```
— returned with HTTP `404`.

## Frontend (UI)

A single-page React dashboard (`Dashboard.jsx`) styled with Tailwind CSS in a dark, "glassmorphism" theme. Key UI pieces:

- **Disclaimer modal** — shown on first visit (tracked via `localStorage`), informing users the predictions are for educational purposes only.
- **Hero section** — product tagline and a "Predict Now" call-to-action button.
- **`StockSelector`** — a searchable custom dropdown of all NIFTY 50 symbols, plus a "Run Prediction" button. Defaults to `RELIANCE.NS` if available.
- **`MetricCard` (×4)** — current price, predicted price, expected % change, and model confidence, each with a loading-skeleton state.
- **`SignalCard`** — a glowing badge showing the BUY/SELL/HOLD recommendation and confidence.
- **`ForecastChart`** — a Recharts area chart visualizing current price vs. predicted price.
- **`AnalysisCard`** — a narrative, plain-English explanation of the model's reasoning, sourced from the API response (with a local fallback generator if missing).
- **Toast notifications** — for surfacing API/network errors.

All API calls are centralized in `src/services/api.js` using Axios, pointed at `VITE_API_BASE_URL` (defaults to `/api`, which Vite's dev server proxies to `http://localhost:3000`).

## Automated Daily Pipeline (GitHub Actions)

`.github/workflows/daily-update.yml` runs the entire data-refresh + retrain pipeline **every day at 6:00 AM IST (00:30 UTC)**, and can also be triggered manually (`workflow_dispatch`):

```yaml
on:
  schedule:
    - cron: '30 0 * * *'   # 00:30 UTC = 06:00 IST
  workflow_dispatch:
```

**Steps performed:**
1. Checks out the repository.
2. Sets up Python 3.12.
3. Installs dependencies from `backend/requirements.txt`.
4. Runs `backend/update.py` to fetch the latest OHLCV data from Yahoo Finance and append it to `backend/Nifty 50.csv`.
5. Runs `backend/train_stocks.py` to re-engineer features, retrain a fresh model for every stock, and regenerate `predictions.json`.
6. Commits and pushes the updated `predictions.json` and `Nifty 50.csv` back to the `main` branch (only if there are actual changes), using the `github-actions` bot identity.

This means the dataset and the model predictions are **self-updating** — no human needs to run the training scripts manually, and the deployed frontend will reflect new predictions every trading day.

## Getting Started Locally

### Prerequisites

- **Node.js** ≥ 18 (the backend `package.json` specifies `node: ^26.3.0` as a dependency pin; any reasonably recent LTS Node version will run the Express server)
- **Python** ≥ 3.10 (the CI workflow uses 3.12)
- **npm** (ships with Node.js)
- **pip**

### 1. Clone the repository

```bash
git clone https://github.com/Surana-Piyush/Stock-Prediction-Platform.git
cd Stock-Prediction-Platform
```

### 2. Backend setup

```bash
cd backend

# Install Node dependencies (Express API server)
npm install

# Install Python dependencies (ML pipeline)
pip install -r requirements.txt
```

**(Optional) Refresh the dataset and regenerate predictions:**

```bash
# Fetch the latest stock data from Yahoo Finance
python update.py

# Engineer features, retrain all 48 models, and write predictions.json
python train_stocks.py

# Or run both steps in one go:
python daily_predictions.py
```

> The first run of `update.py` may take a few minutes since it has to backfill years of history for 48 symbols. `train_stocks.py` also takes a noticeable amount of time since it trains a separate neural network per stock.

**Start the API server:**

```bash
node service.js
# or, with auto-reload during development:
npx nodemon service.js
```

The API will be available at `http://localhost:3000`.

### 3. Frontend setup

In a separate terminal:

```bash
cd frontend
npm install
npm run dev
```

The app will be available at `http://localhost:5173`. Requests to `/api/*` are automatically proxied to `http://localhost:3000` in development (see `vite.config.js`).

### 4. Running everything together

1. Terminal 1 → `cd backend && node service.js`
2. Terminal 2 → `cd frontend && npm run dev`
3. Open `http://localhost:5173` in your browser.

## Deployment

- The **frontend** is deployed on **Vercel**: [stock-prediction-platform-five.vercel.app](https://stock-prediction-platform-five.vercel.app)
- The **backend** needs to be hosted separately (e.g. Render, Railway, a VPS, etc.) since it's a standalone Express server — once deployed, set `VITE_API_BASE_URL` in the frontend's environment to point at the deployed backend's `/api` path.
- The **data + model refresh** runs independently of any hosting provider via the GitHub Actions cron job, committing fresh `predictions.json` straight to the repo — so as long as the backend reads `predictions.json` from the repo (or redeploys on push), predictions stay current automatically.

## Limitations & Known Caveats

- **Not financial advice.** This is a portfolio/educational ML project. Stock prices are influenced by countless factors a purely technical, price-history-based model cannot capture (news, macroeconomics, earnings, sentiment, etc.).
- **`r2` doubles as "Confidence."** Re-using a model fit metric (R²) as a user-facing "confidence" percentage is a simplification — it reflects how well the model fit historical test data, not a true probabilistic confidence in tomorrow's specific prediction.
- **Per-stock models, retrained from scratch daily.** Every stock gets its own independently trained `MLPRegressor`, retrained from a cold start every day — there's no persisted/versioned model artifact, and no walk-forward backtesting beyond the single 80/20 hold-out split.
- **Dependent on Yahoo Finance availability.** `update.py` includes retry/backoff logic specifically because `yfinance` is occasionally rate-limited or returns empty payloads; the daily workflow can silently produce no new rows if Yahoo throttles that day's run.
- **`/api/stocks` is a hardcoded list**, not derived live from the CSV, so adding/removing a tracked symbol currently requires a code change in two places (`stockNames.route.js` and `update.py`'s `NIFTY50` list).
- **No persisted/versioned trained model files** — models exist only transiently in memory during the training script's execution; only their output (`predictions.json`) is persisted.
- **No automated tests** are currently included for either the backend or frontend.
